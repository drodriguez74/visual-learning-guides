The AI
Harness

An AI harness is the structured runtime environment that connects a raw AI model to your tools, context, memory, and workflow — transforming a general intelligence into a specialized, controllable co-worker.

§ 01 What Is a Harness?
Like a horse harness distributes force and gives the rider control without limiting what the horse can do — an AI harness channels the model's raw capability into purposeful, reliable, repeatable behavior. It defines what the AI knows, what it can do, how it should respond, and what constraints it operates under.
FIG. B — HARNESS ANATOMY (3-ZONE MODEL)
Zone A — Input
Human Intent
A developer types a slash command, natural language query, or code snippet
The task carries goal, constraints, and implicit expectations the harness must surface
The harness pre-processes the input — rewriting, augmenting, and injecting context before the model sees it
Examples: /review · /build · a code block · a free-text question
⇅
Harness Bridge
Zone B — Processing
AI Model + Context
The context window is assembled: system prompt + memory + tools + history + retrieved docs + user message
The model reasons within this assembled reality — it can only act on what the harness has placed in context
Tool calls loop back through the harness before the next model turn
Output is raw — unstructured, potentially verbose, unparsed
Zone C — Output
Post-processing & Delivery
Parse Extract structured data from model output — code blocks, JSON, lists, decisions
Validate Schema checks, safety filters, policy enforcement — reject malformed responses
Route Direct output to the right surface: IDE, terminal, PR comment, CI pipeline, webhook
Update Persist new memories, increment version state, trigger downstream agents
§ 02 Anatomy — Layer by Layer
System Prompt
CLAUDE.md · GEMINI.md · copilot-instructions.md
The bedrock instruction set. Defines the AI's role, constraints, output format, coding conventions, and what it must never do. Loaded on every session before any user input. The harness's "job contract" with the model.
ALWAYS LOADED
Agent Persona
agents/*.md · .github/agents/
A Markdown file that injects a specialist role into the model's identity — "you are a senior security engineer reviewing for OWASP Top 10." Loaded per-task, not globally, to maintain context window discipline.
PER-TASK
Skill / Workflow
skills/*/SKILL.md · .github/skills/
Step-by-step procedural guides the model follows before acting. Encodes senior engineering practice — spec-driven dev, TDD, code review protocols. The model consults the skill the way a contractor reads a blueprint.
PER-TASK
Project Context
.ai-sdlc/project-context.md · project-profile.yaml
A structured knowledge base about this specific codebase — language, frameworks, conventions, architecture decisions, team norms, and known constraints. Generated once via /onboard, loaded at session start.
SESSION START
Tool Schemas
tools/ · adapter packs · MCP servers
Function definitions the model can invoke: read a file, run tests, query an issue tracker, call an API, trigger a deployment. The harness resolves tool calls and feeds results back into the context loop.
CAPABILITY LAYER
Conversation History
Session state · summarized prior turns
Prior turns give coherence across a multi-step task. The harness manages truncation — compressing older turns into summaries — to stay within the token budget while preserving key decisions and context.
MANAGED
§ 03 Signal Flow — One Request
FIG. C — A SINGLE /review INVOCATION
1
Developer fires a command
Developer types /review in their AI CLI session. The harness intercepts the raw slash command before it reaches the model.
input: "/review src/auth/login.ts"
↓
2
Harness assembles context
Loads system prompt + security-reviewer agent + code-review skill + project-context.md. Fetches the file contents. Slots each piece into the token-budget-aware context window.
context: [system_prompt, agent, skill, project_ctx, file_content, user_msg]
↓
3
Model reasons + may invoke tools
The model processes the assembled context. If it needs to run tests or read a dependency, it emits a tool call. The harness executes it, appends the result, and sends the model another turn.
tool_call: { name: "run_tests", args: { path: "auth/login.test.ts" } }
↓
4
Output parsed and routed
The harness extracts structured findings from the model's response — severity levels, line references, suggested fixes. Renders them as an inline review comment in the IDE or terminal.
output: { findings: [...], severity: "medium", pr_comment: true }
↓
5
State updated — loop ready
Findings logged to project context. Conversation history compressed and saved. If follow-up is needed, the next turn starts with enriched shared state — the harness gets smarter with each cycle.
state: { memory: updated, history: compressed, next_turn: ready }
§ 04 Three Harness Patterns
🖥️
CLI Harness
Terminal-Native
Instructions live in root markdown files. Commands are slash-prefixed. The AI CLI reads config files from the repo root, linking agents, skills, and project context at session start.
Claude Code Gemini CLI Codex CLI
🧩
IDE Plugin Harness
Editor-Embedded
Lives inside the editor. Reads workspace config, open files, cursor position, and diagnostics. Agents and skills attach as prompt files referenced via inline context handles.
GitHub Copilot Cursor Continue
⚙️
Pipeline Harness
CI/CD-Integrated
Runs as an automated step in a build pipeline. Triggered by PR events, test failures, or deployment hooks. No human in the loop — the harness enforces policy and routes decisions autonomously.
GitHub Actions n8n workflows Agentic CI
§ 05 With a Harness vs. Without
Dimension Without a Harness With a Harness
Context Blank slate every session. No memory of codebase, conventions, or prior decisions. Project context, agent persona, and relevant skills loaded automatically at session start.
Consistency Output quality varies wildly depending on how well the developer prompts each time. Same skill + agent combo fires for the same task type. Reproducible, reviewable results.
Tool Access AI can only suggest code. Cannot run tests, read files, or call APIs without manual copy-paste. Tool schemas define callable actions. Model can read, write, test, deploy — in a controlled loop.
Constraints No guardrails. Model can veer off, hallucinate, or ignore project conventions. System prompt enforces hard constraints. Skills define the process. Policy filters validate output.
Portability Prompts live in developer's head or chat history. Cannot be versioned or shared. Harness is files in source control. Versioned, reviewed, diffable, team-shared.
Multi-Provider Switching AI provider means re-writing all prompts from scratch. Provider-neutral files. Swap CLAUDE.md for GEMINI.md. Assets reuse across harnesses.
A harness doesn't limit what the AI can do — it focuses it.
Raw intelligence becomes reliable craftsmanship.
