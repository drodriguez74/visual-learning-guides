// The 5 Components + Memory

A production loop needs all six. Missing any one and the loop either runs forever, forgets state, or collapses on first failure.

⏰
Automations (The Trigger)
Scheduled cron events, git hooks, CI signals, or Slack messages that start the loop. Without this it's just a one-time run, not a loop.
/loop • scheduled • on-commit • on-fail
🌿
Worktrees (Isolation)
Git worktrees give each parallel agent its own workspace so they don't stomp each other's file writes mid-loop.
git worktree add • parallel safe
📋
Skills (Intent Persistence)
SKILL.md files that encode project conventions, build steps, and "why we don't do it that way." The agent reads this every run so you stop re-explaining it.
SKILL.md • AGENTS.md • project context
🔌
Connectors / MCP
Wire the agent into your real tools — GitHub Issues, Linear, Jira, Slack, Datadog, PagerDuty. The loop reads and writes these as first-class surfaces.
MCP • tool_use • webhooks
🤖
Sub-agents (Maker + Checker)
Split the agent that does the work from the agent that verifies it. The checker subagent is what makes "it's done" mean something.
maker-agent • verifier-agent • orchestrator
💾
Memory (The Critical Layer)
A Markdown file, a GitHub board — anything that lives outside a single conversation. The model forgets everything between runs. The repo does not.
TODO.md • work-log.md • state-on-disk

// G-CAVS — A Framework for Designing Loops

RACE (Role · Action · Context · Expectation) made prompts repeatable. A loop needs its own checklist, because a loop fails in ways a prompt can't: it runs forever, forgets what it did, or mass-produces garbage. G-CAVS is that checklist — five components every production loop needs. Fill in the cards for a loop you actually want to build; the readiness meter rises as your loop becomes well-formed, and the skeleton on the right assembles live.

Try a preset:
G
Goal
A specific, verifiable target — not a vibe.
weak: "improve the code"  ·  strong: names the artifact + the pass condition
C
Context / State
What the agent sees each turn — and where state lives between turns.
weak: "the chat history"  ·  strong: names the on-disk state the loop reloads
A
Action / Tools
The actuators the model can fire — read, write, execute, call.
weak: "let it do stuff"  ·  strong: an explicit, bounded tool list
V
Verification
The hard gate that decides if the last action moved toward the goal.
weak: "looks done"  ·  strong: a mechanical check or strict evaluator — the real bottleneck
S
Stop Conditions
The guardrails that end the loop — success, failure, or escalate.
weak: "until it's fixed"  ·  strong: max-steps + cost cap + no-progress + escalation

Miss G and the loop chases a vibe.
Miss C and it forgets.
Miss A and it can't act.
Miss V and it ships garbage fast.
Miss S and it burns your budget.

// The Ralph Loop — Where It Started

    

      Geoffrey Huntley's "Ralph" pattern (July 2025) is the conceptual seed of modern loop engineering. Named after Ralph Wiggum — deterministically simple in an unpredictable world.
    


    

      
The non-obvious insight: context reset per iteration. Long sessions degrade as the window fills with dead ends. Ralph sidesteps this by starting fresh each tick, reading current state from disk.

      
#!/bin/bash  # The Ralph Loop — one unit of work per tick
MAX=10
for i in $(seq 1 $MAX); do

  # 1. TRIGGER — feed spec to fresh agent context
  cat PROMPT.md | claude -p --dangerously-skip-permissions

  # 2. VERIFY — check exit conditions
  grep -q "BLOCKED" specs/TODO.md && break
  grep -q "\[ \]" specs/TODO.md || break

  # 3. MEMORY — agent committed its work; repo holds state
  sleep 10

done
# The model forgets. The repo doesn't.

    


    

      Modern loop engineering (Claude Code's /loop, Codex Automations) is this same pattern elevated to a product primitive — with built-in worktrees, skill loading, and sub-agent orchestration.
    

  







  

    
// Stopping Conditions — The Most Important Design Decision

    

      A loop without explicit termination logic is the single most common — and most expensive — mistake. Design your stop conditions before you write a single loop instruction.

// Prompt vs. Loop — What Actually Changes


    

      
        

          Dimension
          Prompt Engineering (2025)

          Typing each turn, in the driver's seat

          One loop definition → N autonomous iterations

          You read the output and judge
          Verifier sub-agent + mechanical checks

          When a defined goal condition holds

          Mass-produced mistakes if verifier is weak

          Low — you stay in the loop
          Comprehension debt grows if you don't read output
        

      
    


    
// Comprehension Debt — The Hidden Cost

    
The faster the loop ships code you didn't write, the bigger the gap between what exists and what you actually understand. Designing the loop with judgment is the cure; designing it to avoid thinking is the accelerant.
