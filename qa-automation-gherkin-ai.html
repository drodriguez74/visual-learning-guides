02
LanguageGherkin Syntax & Keywords
Feature:
Names the feature being tested. One per file. Sets the context for all scenarios below it.
Scenario:
A single test case. Each Scenario is independent and should be able to run in isolation.
Given
Sets up the pre-condition / initial state before the action is performed.
When
The action or event the user performs. The trigger. Usually one per scenario.
Then
The expected outcome. What the system should show or do after the When action.
GHERKIN · Feature File login.feature
@smoke @authentication Feature: User Login # Background runs before EVERY scenario in this file Background: Given the application is running at "https://app.example.com" Scenario: Successful login with valid credentials Given the user is on the login page When the user enters username "darwin@example.com" And the user enters password "SecurePass123" And the user clicks the "Sign In" button Then the dashboard page is displayed And the welcome message contains "Darwin" Scenario: Login fails with invalid password Given the user is on the login page When the user enters username "darwin@example.com" And the user enters password "WrongPass" And the user clicks the "Sign In" button Then an error message "Invalid credentials" is displayed Scenario Outline: Login with multiple user roles Given the user logs in as "<role>" with password "<password>" Then they see the "<expectedPage>" page Examples: | role | password | expectedPage | | admin | Admin123 | admin-panel | | viewer | View456 | read-only | | editor | Edit789 | workspace |
03
ImplementationStep Definitions — The Bridge
STEP DEFINITIONS · Java + Selenium LoginSteps.java
import io.cucumber.java.en.*; import org.openqa.selenium.*; import org.openqa.selenium.chrome.*; public class LoginSteps { // ① WebDriver injected via @Before hook private WebDriver driver; // ② Regex pattern MUST match Gherkin sentence exactly @Given("the user is on the login page") public void navigateToLoginPage() { driver.get("https://app.example.com/login"); } // ③ Capture parameter from Gherkin with {string} @When("the user enters username {string}") public void enterUsername(String username) { driver.findElement(By.id("username")) .sendKeys(username); } @When("the user enters password {string}") public void enterPassword(String password) { driver.findElement(By.id("password")) .sendKeys(password); } @When("the user clicks the {string} button") public void clickButton(String buttonText) { driver.findElement( By.xpath("//button[text()='"+buttonText+"']") ).click(); } // ④ Assertion in Then step @Then("the dashboard page is displayed") public void verifyDashboard() { // ⑤ Assert with org.junit.Assert or assertj assert driver.getCurrentUrl().contains("/dashboard"); } @Then("the welcome message contains {string}") public void verifyWelcome(String name) { String msg = driver .findElement(By.cssSelector(".welcome-header")) .getText(); assert msg.contains(name); } }
① Shared Driver
WebDriver is injected via Cucumber hooks (@Before/@After), not instantiated per method. Shared state between steps in a scenario.
② Regex / Cucumber Expression
The string in @Given/@When/@Then MUST exactly match the Gherkin sentence. Cucumber maps them at runtime via regex.
③ Parameter Capture
{string} captures the quoted value from Gherkin and passes it as a method argument. Also supports {int}, {float}, {word}.
④ Assertions in @Then
Then steps verify the expected outcome. Use JUnit Assert, AssertJ, or Hamcrest for expressive failures.
⑤ Page Object Model
Best practice: wrap findElement calls in Page Object classes. Step Defs call methods, not raw Selenium. AI generates both simultaneously.
04
AI AccelerationNew UI vs. Existing UI
The strategies are fundamentally different. New UI (zero automation) means AI generates everything from scratch — features, step defs, page objects, and the framework itself. Existing UI means AI analyzes and extends what's already there without breaking existing tests.
🆕
New UI — Zero Automation
Greenfield · Build from scratch
1
Scaffold the framework
Paste your tech stack (Java/Selenium/Cucumber/Maven). AI generates pom.xml, folder structure, cucumber.properties, and base runner in one shot.
# Prompt AI:
"Generate a Maven project structure
for Cucumber + Selenium + Java with
Page Object Model, ExtentReports,
and parallel execution support."
2
Generate feature files from specs
Feed AI the user story or acceptance criteria. It outputs a complete .feature file with multiple scenarios including edge cases you may have missed.
# Prompt AI:
"Convert this Jira story to a Gherkin
feature file with happy path, error
cases, and a Scenario Outline for
multiple user roles: [paste story]"
3
Generate Step Defs + Page Objects
Paste the .feature file. AI generates matching Step Definition class AND the Page Object class together. Both are wired to each other correctly.
# Prompt AI:
"Generate Java Step Definitions AND
a Page Object class for these Gherkin
steps using Selenium + WebDriverManager:
[paste feature file]"
4
Generate locators from screenshots
Screenshot the UI → ask AI to identify the best CSS selectors, IDs, or XPaths for each element. Huge time saver when dev hasn't added test IDs yet.
# Prompt AI (with screenshot):
"Identify robust CSS selectors for
the login form elements in this UI
screenshot. Prefer data-testid,
then id, then CSS, avoid XPath."
5
Generate test data + hooks
AI generates @Before/@After hooks, test data factory classes, and environment config files. Full setup in minutes not days.
~70%
Faster than manual. Framework + first 20 scenarios in a day instead of a week.
🔄
Existing UI — Partial Automation
Brownfield · Extend & modernize
1
Audit existing coverage
Paste your existing feature files to AI. Ask it to identify gaps — missing edge cases, uncovered user flows, scenarios with no negative tests.
# Prompt AI:
"Review these existing Gherkin
scenarios. Identify gaps: missing
edge cases, error scenarios, and
boundary conditions. [paste files]"
2
Extend without breaking
Feed AI an existing Step Definition class + a new .feature file. It adds new methods while matching your existing code style, naming conventions, and Page Object pattern.
# Prompt AI:
"Add step definitions for these new
Gherkin steps. Match the pattern of
the existing class below. Reuse
existing page objects where possible:
[paste existing steps + new gherkin]"
3
Fix flaky tests with AI diagnosis
Paste failing test output + the step definition. AI identifies the root cause (missing waits, stale element, timing issue) and provides the fix.
# Prompt AI:
"This test is flaky. Here's the
exception stack trace and step def.
Diagnose the root cause and provide
a fixed version: [paste error + code]"
4
Refactor step definitions
Legacy automation often has duplicated steps, hardcoded values, and no Page Object layer. AI refactors the whole class to POM pattern in one pass.
# Prompt AI:
"Refactor these step definitions to
use Page Object Model. Extract all
locators and actions to a separate
Page class. Keep Gherkin unchanged."
5
Generate scenarios for new features
When a new feature lands on an existing app, AI generates new scenarios in the same style as existing ones. Consistency is automatic.
~50%
Faster coverage expansion. New features automated same-sprint instead of backlogged.
