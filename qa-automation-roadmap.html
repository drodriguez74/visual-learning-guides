BDD + POM in Practice

Gherkin → Steps → POM

The complete traceability chain: a business-readable Gherkin scenario maps to step definitions that delegate to Page Objects or API clients. AI agents keep all three layers in sync as the application evolves.

UI Test — Gherkin Feature → Step → POM ChainTypeScript · Playwright · POM
① Feature File
# features/ui/login.feature Feature: User Login @ui @smoke Scenario: Successful login Given user is on login page When enters valid credentials Then dashboard is displayed @ui @regression Scenario Outline: Failed login Given user is on login page When enters <user> / <pass> Then error <msg> is shown Examples: | user | pass | msg | | bad | bad | Invalid | | empty | empty | Required |
② Step Definition
// steps/login.steps.ts import { Given, When, Then } from '@cucumber/cucumber'; import { LoginPage } from '../pages'; import { World } from '../support/world'; Given('user is on login page', async function(this: World) { this.loginPage = new LoginPage(this.page); await this.loginPage.open(); }); When('enters valid credentials', async function(this: World) { await this.loginPage .fillUsername(config.user) .fillPassword(config.pass) .submit(); }); Then('dashboard is displayed', async function(this: World) { const dash = new DashboardPage(this.page); await dash.assertLoaded(); });
③ POM Class
// pages/LoginPage.ts import { Page, Locator } from '@playwright/test'; import { BasePage } from './BasePage'; export class LoginPage extends BasePage { // Playwright locators // (AI self-heals these) readonly username: Locator; readonly password: Locator; readonly submitBtn: Locator; constructor(page: Page) { super(page); this.username = page.getByLabel('Username'); this.password = page.getByLabel('Password'); this.submitBtn = page.getByRole('button', {name: 'Log in'}); } async open() { await this.page.goto( config.loginUrl); } async submit() { await this.submitBtn.click(); } }
API Test — Gherkin Feature → Step → axios ClientTypeScript · axios · zod
① API Feature File
# features/api/disputes.feature Feature: Disputes API Background: Given API is authenticated And base URL is configured @api @smoke Scenario: Create dispute returns 201 When POST /disputes with valid payload Then response status is 201 And response has disputeId And response matches schema @api @regression Scenario: Get dispute by ID Given a dispute exists with id "D-001" When GET /disputes/D-001 Then status is 200 And field status equals "OPEN"
② API Step + axios Client
// steps/disputes.api.steps.ts import { When, Then } from '@cucumber/cucumber'; import { disputeClient } from '../api/disputeClient'; import { DisputeSchema } from '../schemas/dispute'; When('POST /disputes with valid payload', async function(this: World) { this.response = await disputeClient.create( DisputeFactory.valid() ); }); Then('response status is {int}', async function( this: World, code: number) { expect(this.response.status) .toBe(code); }); Then('response matches schema', async function(this: World) { // zod schema validation DisputeSchema.parse( this.response.data); });
Language Decision — Why This StackAgreed Architecture

"Each job in the pipeline uses the best language for its job. Languages share nothing but artifacts."

LayerLanguageKey LibrariesRationale
Step Definitions TypeScript @cucumber/cucumber · ts-node Already your codebase. Strong typing catches POM errors at compile time.
POM Classes TypeScript @playwright/test · Locator API Playwright is TypeScript-native. getByRole/getByLabel are more resilient than XPath.
API Test Clients TypeScript axios · zod · faker-js axios + zod gives typed requests and schema validation in the same ecosystem.
AI Agents Python 3.12 gateway-client · gitpython · bs4 Best AI/ML library ecosystem. The model is called through the platform gateway (provider-agnostic — GPT-5.2 / Sonnet 4.6 bake-off pending); DOM parsing first-class.
Reporting / Email Python 3.12 jinja2 · smtplib · allure-python 20-line email sender + HTML template. No framework needed. Simple and debuggable.
CI Glue bash (minimal) echo · cp · if/else only Max 10 lines per job. Anything more complex goes into Python. Shell = wiring only.
❌ Java / Selenium Not used — Team is JS/TS. Migration cost unjustified. Playwright supersedes Selenium for new UI tests.
Key insight: Languages never cross job boundaries in GitLab CI — only artifacts do. The TypeScript test jobs output allure-results/ JSON. The Python report job reads that JSON and sends the email. No shared runtime, no dependency conflicts, each image stays lean.

Pipeline as Code

GitLab CI — Golden Path

Every merge request triggers the feature test suite. Full regression runs twice a week on a cron schedule. Reports are emailed automatically. Each pipeline job uses its own Docker image — Playwright for UI/API tests, Python for AI agents and reporting. No shared runtime required. This is the single-repo view; the company-wide enforcement model (shared template, coverage gates, MR gate semantics via QA_TARGET_MODE, breadcrumb checks) is specified in the QA Evergreen Ecosystem design.

AI Acceleration

AI Agents at Every Layer

AI isn't a bolt-on — it's woven through every layer of the framework. These are the specific AI patterns that make the system 10× faster to build and self-sustaining to maintain.

🔍
Scope Detector Agent
Reads git diff on every MR. Maps changed files (controllers, services, components) to their corresponding Cucumber @tags. Injects TEST_TAGS variable into GitLab CI — so only affected tests run. No manual annotation.
📝
Feature File Generator
Reads OpenAPI specs and user stories. Generates Gherkin feature files with Given/When/Then scenarios covering happy paths, edge cases, and negative tests. SDETs review, refine, and commit.
🏗️
Step Definition Scaffolder
Given a new .feature file, generates a matching TypeScript step definition skeleton with correct @cucumber/cucumber imports, World context typing, and wiring to POM or API client classes.
🖥️
POM Generator
Crawls application pages using a headless browser, extracts all interactive elements, and generates Page Object Model classes with locators, action methods, and verification helpers.
🔧
Self-Healing Locator Agent
When a selector fails at runtime, the agent first replays the step's Gherkin intent via agent-browser — if the intent also fails, it files a defect instead of healing. If the intent succeeds, it finds the new element in the accessibility-tree snapshot, patches the POM class, and opens a PR. Auto-merge requires a corroborating breadcrumb drift event — unexplained DOM changes stay red.
📊
Report Narrator Agent
After each run, reads test results and produces a plain-English summary: which tests failed, why (correlated to recent commits), what modules are affected, and what action to take — sent in the email report.
🧹
Coverage Audit Agent
Weekly: scans all API routes and UI components against the test suite. Identifies uncovered paths and opens Jira tickets with auto-generated feature file drafts attached for SDET pickup.
🚫
Flakiness Quarantine Agent
Monitors test pass/fail history across runs — including retried-then-passed results, which count as flakes. Tests failing intermittently >3 times are auto-tagged @flaky, isolated into a quarantine suite, and a Jira investigation ticket is auto-created.
🌐
Agent-Browser Verification Crawl
Weekly cron: the agent-browser CLI logs in as each role (encrypted auth vault), walks every route, and diffs the live accessibility tree against the breadcrumb inventory — catching feature-flagged pages, role-access mismatches, and unrendered actions that static analysis can't see. Fenced by --allowed-domains and --action-policy; never on the blocking MR path.
🧭
Journey Discoverer & Trace Compiler
Journeys are demonstrated, not imagined. QA writes a plain-English scenario (or the explorer proposes one), the agent executes it in a real browser, and every action is recorded — accessibility role + name, URL, API calls observed in the HAR. The trace compiles into a deterministic .feature + step definitions + POM with semantic locators (never session refs). Discovery is expensive once; replay is deterministic Playwright forever. For CMS-built workflow apps (disputes, chargebacks, money movement), the config export seeds the inventory the agent then verifies.
