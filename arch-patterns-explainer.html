Pattern

Layered Architecture

Also called N-Tier. Organizes code into horizontal layers where each layer has a distinct responsibility and only communicates with the layer directly beneath it.

Layered / N-Tier Monolithic · Structured

"Separation of concerns, top to bottom."

Each layer has a single responsibility. The Presentation layer handles I/O, Business Logic enforces rules, Persistence manages data access, and the Database stores it. Changes in one layer don't cascade — in theory.

✓ Strengths
  • Easy to understand & onboard
  • Clear ownership per layer
  • Testable in isolation
  • Dominant in enterprise Java/Spring
✗ Weaknesses
  • Tight coupling over time
  • Hard to scale independently
  • "Monolith trap" as app grows
  • Slow end-to-end changes
Used In
Spring Boot Apps Banking Systems CMS Platforms Internal Tools
02

Pattern

Microservices

Decomposes an application into small, independently deployable services each owning a bounded context, communicating over APIs or messaging.

Microservices Architecture Distributed · Independent

"Small teams, small services, big scale."

Services are independently deployable, technology-agnostic, and communicate via REST, gRPC, or events. Each service owns its bounded context including its database — preventing shared-data coupling.

✓ Strengths
  • Independent scaling & deploy
  • Technology heterogeneity
  • Fault isolation
  • Team autonomy
✗ Weaknesses
  • Distributed system complexity
  • Network latency & failures
  • Harder to trace & debug
  • Operational overhead
Used In
Netflix Amazon Uber Fiserv Platforms
03

Pattern

Event-Driven Architecture

Components communicate exclusively through events — producers emit, consumers react. Systems remain fully decoupled: producers don't know who is listening.

Event-Driven Async · Decoupled · Reactive

"Emit and forget. React and adapt."

A producer publishes an event to a broker (Kafka, RabbitMQ, SQS). Zero or many consumers subscribe. The producer is completely unaware of downstream behavior — achieving maximum decoupling.

✓ Strengths
  • Zero producer-consumer coupling
  • Highly scalable & resilient
  • Audit log via event history
  • New consumers = zero producer change
✗ Weaknesses
  • Eventually consistent
  • Hard to trace event flows
  • Schema evolution is complex
  • Ordering guarantees tricky
Used In
Fraud Detection Real-time Analytics Dispute Processing IoT Pipelines
04

Pattern

CQRS + Event Sourcing

Command Query Responsibility Segregation — separate the write model (commands that change state) from the read model (queries that return data). Often paired with Event Sourcing.

CQRS + Event Sourcing Write/Read Split · Audit-First

"Writes store facts. Reads show views."

Commands mutate state and produce events stored in an immutable log. Projections consume those events to build optimized read models. You can replay history to rebuild any view — or create new ones retroactively.

✓ Strengths
  • Full audit history built-in
  • Read/write scaled independently
  • Time travel via event replay
  • Complex domain modeling
✗ Weaknesses
  • High conceptual complexity
  • Eventual read consistency
  • Schema migration is hard
  • Overkill for simple CRUD
Used In
Financial Ledgers Dispute Systems Inventory Healthcare Records
05

Pattern

Hexagonal (Ports & Adapters)

Isolates the core domain from infrastructure. The domain only knows about ports (interfaces); adapters implement them — making the core swappable and infinitely testable.

Hexagonal / Ports & Adapters Clean Architecture · Domain-Centric

"The domain is the sun. Infrastructure orbits it."

The domain defines ports (interfaces). Adapters implement them. A REST controller and a test harness are both just adapters. Infrastructure — DBs, queues, third-party APIs — is swappable without touching domain logic.

✓ Strengths
  • Domain fully testable in isolation
  • Infrastructure is interchangeable
  • Forces clean boundaries
  • Scales across teams cleanly
✗ Weaknesses
  • More interfaces & boilerplate
  • Steeper learning curve
  • Over-engineered for small apps
  • Verbose adapter layer
Used In
DDD Applications Clean Architecture Spring Boot Testable Services
06

Pattern

Saga Pattern

Manages long-running distributed transactions across microservices through a sequence of local transactions — each with a compensating transaction that can undo it on failure.

Saga (Choreography & Orchestration) Distributed Tx · Compensation

"If any step fails, undo every step before it."

Distributed transactions can't use two-phase commit at scale. Sagas replace this with a chain of local transactions. Each step publishes an event; if a downstream step fails, compensating transactions roll back prior steps.

✓ Strengths
  • Distributed tx without 2PC
  • Service independence maintained
  • Resilient to partial failures
  • Explicit compensation logic
✗ Weaknesses
  • Compensation logic is complex
  • Eventual consistency only
  • Hard to test all failure paths
  • Debugging is non-trivial
Used In
E-Commerce Checkout Payment Processing Dispute Workflows Travel Booking

Pattern Comparison

Choose based on team size, scale requirements, consistency needs, and domain complexity.

Pattern Best For Consistency Complexity Scale
Layered Small–medium monoliths, CRUD apps Strong (single DB) Low Vertical only
Microservices Large teams, independent scaling Eventually consistent High Horizontal per service
Event-Driven Real-time, async, reactive pipelines Eventual Medium–High Excellent
CQRS + ES Audit-heavy, complex domain, high read load Eventual (reads) Very High Read/write independent
