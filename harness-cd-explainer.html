Security Architecture

The Delegate

The Delegate is Harness's secret weapon for security. It's a lightweight agent you run inside your own infrastructure. It phones home to the Harness control plane — meaning no inbound firewall holes, no VPN tunnels, no credentials stored outside your trust boundary.

Delegate Architecture — Outbound-Only Security ModelNo Inbound Ports

"Outbound only. Your perimeter stays sealed."

Traditional SaaS CD tools require you to open inbound ports or grant them VPN access — a huge security risk. The Delegate flips this: it lives inside your VPC and polls Harness for work. Harness never needs to reach into your network.

What Delegate Does
  • Polls Harness for pending tasks
  • Executes kubectl / helm / terraform
  • Pulls artifacts from registries
  • Resolves secrets from Vault / AWS SM
  • Sends results back to Harness
  • Reports health metrics to CV
Deployment Options
  • Kubernetes pod (most common)
  • Docker container
  • Shell script on a VM
  • Helm chart install
  • ECS task definition
  • Multiple delegates per env

Execution Model

Pipeline Anatomy

A Harness pipeline is a YAML-defined, visually rendered sequence of Stages. Each Stage targets a Service + Environment + Infrastructure combination. Steps within stages are policies, scripts, approvals, or native integrations — no custom glue code needed.

Typical CD Pipeline — Code to ProductionStage Flow
Pipeline Core Concepts
Service
The deployable unit — a microservice, job, or application. Defines artifact sources (Docker image, Helm chart, JAR file), configuration, and service-level variables. Services are versioned and reusable across environments.
Artifact · Manifest · Config
Environment
A logical deployment target — dev, staging, prod, or any custom label. Holds environment-specific overrides for config, secrets, and variables. Supports environment groups for multi-region fan-outs.
Dev · Staging · Prod · DR
Infrastructure Definition
Specifies where the deployment runs — a Kubernetes cluster namespace, an ECS cluster, a set of VMs. Connects the environment to the underlying compute via the Delegate.
K8s · ECS · VM · Serverless
Execution Steps
The actual work: rolling deploy, canary weight shift, helm upgrade, terraform apply, HTTP call, shell script, approval wait, or a Verify step that pauses the pipeline until metrics confirm health.
Deploy · Verify · Approve · Notify
Templates
Reusable building blocks at step, stage, or full pipeline level. Platform teams encode best practices once; dev teams instantiate them. Updates to a template auto-propagate to all pipelines that reference it — solving pipeline sprawl at scale.
Step · Stage · Pipeline · Service Templates

Progressive Delivery

Deployment Strategies

Harness ships four battle-tested deployment strategies out of the box — no scripting required. Choose based on your risk tolerance, traffic patterns, and infrastructure type.

🔄
Rolling Low Risk
Updates pods in batches. Always running, minimal downtime. Best for stateless services where brief mixed-version state is acceptable. Simple to configure and reason about.
🐤
Canary Safest
Routes a small % of traffic to the new version. CV monitors metrics at each weight step. Bad canary? Auto-rollback before most users are affected. Harness controls traffic weights without scripting.
🔵🟢
Blue-Green Zero Downtime
Two identical environments run in parallel. v2 is fully tested in Green before traffic switches. Rollback is instant — flip the load balancer back. No mixed versions, no downtime.
⚙️
Custom Any Pattern
Mix-and-match steps to build any deployment pattern. Chain shell scripts, Terraform, Ansible, Helm, HTTP calls, and ServiceNow integrations into a single cohesive stage.

AI Safety Net

Continuous Verification

CV is Harness's ML-powered deployment watchdog. It queries your APM tools during and after deployment, uses machine learning to baseline normal behavior, and automatically rolls back the pipeline if anomalies exceed a configurable risk threshold.

Continuous Verification FlowML-Powered · Auto-Rollback

"Deploy safely. Let ML decide if it's healthy."

After each deployment step, Harness pauses the pipeline and runs a Verify step. It samples metrics and logs, compares them to a learned baseline from previous healthy deployments, and computes a risk score. If the score crosses the configured threshold, rollback fires automatically — before your users notice.

What CV Monitors
  • Error rate / exception count
  • Response time / latency p99
  • Throughput / RPS
  • Custom metrics (any APM)
  • Log patterns / anomalies
  • Deployment health score
