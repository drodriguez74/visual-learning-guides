02Recipe: Extracting User Journeys

Before a single line of Gherkin, you decide what to automate. Point this six-question recipe at any UI — new or existing — and it yields a prioritized, decomposed set of journeys ready to become feature files. This is the step most teams skip, and it's why suites become brittle and bloated.

1
Inventory the surface WHAT exists?
Walk every screen and list each page, modal, and state. For an existing app, crawl the sitemap or route table. Output: a flat list of every place a user can be. AI can parse a sitemap, route config, or screenshots into this inventory instantly.
2
Identify personas & roles WHO uses it?
List each distinct user type — admin, standard, guest, read-only. Each persona has different journeys and permissions. This directly seeds your Scenario Outline Examples tables later.
3
Map the critical paths WHAT delivers value?
For each persona, trace the end-to-end flows that produce real outcomes: "log in → search → add to cart → checkout". These are journeys, not pages. A journey crosses multiple screens toward a goal.
4
Score by risk × frequency WHAT first?
Don't automate everything. Rank each journey on how often it runs and how costly its failure is. High-risk + high-frequency journeys get automated first (see the matrix below). This keeps the suite lean and high-value.
5
Decompose into atomic steps HOW does it flow?
Break each journey into discrete user actions and expected outcomes. Each becomes a Given / When / Then line. Tag steps that recur across journeys — those become shared, reusable step definitions.
6
Define data & preconditions WHAT state?
Each journey needs test data (users, products, fixtures) and a starting state. Capture these now so your Background blocks and data factories write themselves. Output of this recipe = a ready-to-write feature backlog.
📐 Step 4 — Risk × Frequency Prioritization Matrix
HIGH RISK · HIGH FREQUENCY
→ Automate first (P1)
Login, checkout, payment, core CRUD. Run on every commit. Your smoke + regression backbone.
HIGH RISK · LOW FREQUENCY
→ Automate next (P2)
Account deletion, refunds, admin overrides. Rare but catastrophic if broken. Nightly runs.
LOW RISK · HIGH FREQUENCY
→ Automate selectively (P3)
Search filters, sorting, navigation. Automate the happy path; skip exhaustive variants.
LOW RISK · LOW FREQUENCY
→ Leave manual (P4)
Cosmetic settings, edge tooltips. Manual or exploratory testing. Don't pay the maintenance tax.
03Framework Structure (Maven + Playwright + Cucumber)

A clean separation of concerns is what keeps a suite maintainable. Features describe behaviour, step definitions translate it, page objects own all the Playwright selectors. Nothing in a step definition ever touches a raw selector.

05Layer 2 — Step Definitions (the glue)

Each annotation string matches a Gherkin line exactly; {string} captures the quoted value as a parameter. Notice the steps contain no selectors — they delegate to page objects. That's what makes them survive UI redesigns.

javaCheckoutSteps.java
Java · Cucumber-JVM
package com.fiserv.qa.stepdefs;import io.cucumber.java.en.*;import com.fiserv.qa.pages.CheckoutPage;import com.fiserv.qa.support.World;public class CheckoutSteps {  // World holds the shared Playwright Page for this scenario  private final CheckoutPage checkout;  public CheckoutSteps(World world) {    this.checkout = new CheckoutPage(world.page);  }  @When("they add {string} to the cart")  public void addToCart(String product) {    checkout.addItem(product);   // page object does the work  }  @When("they submit payment {string}")  public void submitPayment(String cardNumber) {    checkout.payWith(cardNumber);  }  @Then("an order confirmation number is displayed")  public void verifyConfirmation() {    checkout.assertConfirmationVisible();  }  @Then("they see the error {string}")  public void verifyError(String message) {    checkout.assertErrorText(message);  }}
06Layer 3 — Playwright Page Object

All selectors live here. Playwright's killer feature: locators auto-wait for elements to be visible, stable, and actionable — so you almost never write explicit waits, which is the #1 source of flaky Selenium tests. Prefer role-, label-, and test-id-based locators over CSS/XPath; they're far more resilient to redesigns.

javaCheckoutPage.java
Java · Playwright
package com.fiserv.qa.pages;import com.microsoft.playwright.*;import com.microsoft.playwright.options.AriaRole;import static com.microsoft.playwright.assertions              .PlaywrightAssertions.assertThat;public class CheckoutPage {  private final Page page;  // Locators defined once — resilient, role/test-id based  private final Locator confirmation;  private final Locator errorBanner;  public CheckoutPage(Page page) {    this.page = page;    this.confirmation = page.getByTestId("order-confirmation");    this.errorBanner  = page.getByRole(AriaRole.ALERT);  }  public void addItem(String product) {    // auto-waits until the button is actionable — no sleep()    page.getByRole(AriaRole.LISTITEM,         new Page.GetByRoleOptions().setName(product))        .getByRole(AriaRole.BUTTON,         new Locator.GetByRoleOptions().setName("Add"))        .click();  }  public void payWith(String cardNumber) {    page.getByLabel("Card number").fill(cardNumber);    page.getByRole(AriaRole.BUTTON,         new Page.GetByRoleOptions().setName("Pay now"))        .click();  }  public void assertConfirmationVisible() {    assertThat(confirmation).isVisible();  // auto-retries  }  public void assertErrorText(String expected) {    assertThat(errorBanner).hasText(expected);  }}
07Two Starting Points, One Destination

Whether you're standing up automation from zero or extending a mature suite, the recipe is the same — only the AI strategy differs. Greenfield generates; brownfield assimilates.

2
Scaffold the framework

3

~70%

🏗️
Brownfield — Existing Suite
// some automation already exists
1
Extract the existing patterns
Give AI 2–3 existing files → it derives your conventions into a reusable style guide.
2
Audit coverage against the recipe
Map existing features to the journey backlog → AI flags the gaps to fill.
3
Generate additively, in-style
New steps reuse existing page objects and match naming — nothing existing is touched.
4

08The AI Build-Skill — Reusable Prompt Chain

Run these in sequence to take any UI from zero to a working suite — or extend an existing one. The trick for consistency is Prompt 0: extract your house style once, then prepend it to every later prompt so AI always generates code that looks like your code.

🧩
How to use this as a skill: save these prompts as a workflow file in your AI-SDLC toolkit. Prompt 0 produces a STYLE.md you paste into every subsequent prompt's context. Greenfield starts at Prompt 1; brownfield starts at Prompt 0.

I will paste it into future prompts.
