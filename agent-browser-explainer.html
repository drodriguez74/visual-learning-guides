Agent Thoughts

01 — Overview

What is an Agent Browser?

A software agent that operates a real web browser — clicking, scrolling, typing, and navigating — driven by an LLM that perceives the page and decides the next action.

02 — Core Loop

The Perceive–Reason–Act Loop

Every agent browser turn follows the same tight loop — until the goal is satisfied or it hits a dead end.

STEP 01
👁️
Observe
Agent receives a screenshot and/or accessibility tree (DOM) of the current page state. It sees what a human would see.
STEP 02
🧠
Reason
The LLM reads the observation, compares to the goal, recalls prior steps, and reasons about what action to take next.
STEP 03
⚡
Act
Issues a structured tool call: click, type, scroll, navigate, or extract. The browser executor carries it out.
STEP 04
🔁
Evaluate
The new page state is captured. The agent checks if the goal is reached, if an error occurred, or if replanning is needed.

03 — Perception

How the Agent Sees the Web

Browsers don't hand agents clean data. Perception is a translation layer between raw HTML and LLM-readable meaning.

Raw HTML / DOM
Full page source, deeply nested, noisy, verbose
→
Accessibility Tree
Roles, labels, interactive elements extracted
→
Screenshot
Visual context; vision-language model reads layout
→
State Summary
Compressed context passed to LLM with history
→
LLM Input
Goal + state + memory → next action token

04 — Toolset

What the Agent Can Do

The agent operates through a constrained set of browser primitives — the same actions a human hand and eye perform, expressed as tool calls.

click(selector)
Clicks any element — buttons, links, checkboxes, dropdowns. Can target by CSS selector or coordinate.
type(text)
Fills form fields, search boxes, text areas. Can clear existing value first.
navigate(url)
Goes directly to a URL, or uses back/forward in browser history.
scroll(direction)
Scrolls the viewport or a container up, down, left, right — reveals off-screen content.
screenshot()
Captures the current viewport as an image for visual reasoning and progress verification.
extract(query)
Reads and returns structured data from the page — tables, prices, text content.
wait(condition)
Pauses until an element appears, a URL changes, or a network request completes.
select(option)
Picks from dropdown menus, radio groups, or multi-select lists by value or label.

05 — Applications

Where Agent Browsers Operate

Anywhere a browser is the interface — whether public web or internal enterprise tools — an agent browser can take over repetitive, multi-step workflows.

🧪 QA Automation
Executes end-to-end test scenarios against live apps, adapting to dynamic UI changes without brittle selectors.
🔍 Web Research
Navigates multiple sites, extracts structured data, synthesizes results — like a human analyst, but at scale.
🛒 Workflow Automation
Books travel, fills procurement forms, submits tickets — operating legacy enterprise UIs that have no API.
🤖 Agentic Assistants
Delegates sub-tasks to browser agents while a higher-level orchestrator coordinates toward a complex goal.

06 — Challenges

Hard Problems

Agent browsers are powerful but fragile. Understanding their failure modes is essential before production deployment.

🎯
Selector Brittleness
Dynamic class names, shadow DOM, and framework-generated IDs break tool calls that rely on CSS selectors.
🔄
Context Blowout
Long browsing sessions eat the LLM's context window fast — screenshots and DOM trees are expensive tokens.
🧩
CAPTCHAs & Auth Walls
Human-verification challenges and SSO flows can block the agent entirely with no programmatic bypass.
⚠️
Irreversible Actions
Clicking "Submit Order" or "Delete Account" has real consequences. Agents need explicit checkpointing and human-in-the-loop gates for high-stakes actions.
🌀
Infinite Loops
Agents can get stuck in navigation cycles — paginated results, modal dialogs, or error states that look like progress.
🐢
Latency Cost
Each action involves a screenshot + LLM inference + tool execution round-trip. Complex tasks can take minutes and significant token spend.
