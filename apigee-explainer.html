Overview

What is Apigee?

Apigee is Google Cloud's full-lifecycle API management platform. It acts as a programmable reverse proxy: every API call passes through Apigee where security, transformation, routing, caching, and analytics policies execute before the request ever reaches your backend.

Configuration Store

Key Value Maps (KVM)

KVMs are Apigee's secure, runtime configuration store. Instead of hardcoding backend URLs, API keys, feature flags, or credentials inside proxy policies, you store them in an encrypted KVM and read them at runtime — enabling zero-redeployment config changes.

Broadest
01
Organization
Scope · All Environments

Shared across every environment (dev, staging, prod) in your Apigee org. Ideal for global constants like third-party vendor URLs or shared encryption keys.

Per-Env

Scoped to a specific environment. Use for backend URLs that differ between dev/staging/prod, or env-specific rate limit thresholds and feature flags.

Scope · Single Proxy

Accessible only within a single API proxy. Perfect for proxy-specific secrets or configuration that must never leak into other proxies in the same environment.

KVM at Runtime — How it works KeyValueMapOperations Policy

"Config lives outside the proxy. No redeploy needed."

The KeyValueMapOperations policy is attached to the PreFlow. On the first request it reads the KVM, caches the values for N seconds (TTL), and stores them as flow variables. All subsequent policies in the pipeline can reference those variables without re-querying the store.

What to Store
  • Backend service URLs
  • OAuth client credentials
  • 3rd-party API keys
  • Feature flags
  • Rate limit thresholds
  • IP allowlists
Best Practices
  • Read in PreFlow only
  • Set ExpiryTimeInSecs
  • Always set fallback values
  • Avoid writes in hot path
  • Use Env scope for URLs
  • Org scope for global keys
Sample Policy XML
<!-- KeyValueMapOperations policy --> <KeyValueMapOperations name="KVM-ReadConfig" mapIdentifier="env-config"> <ExpiryTimeInSecs>300</ExpiryTimeInSecs> <Get assignTo="kvm.backend_url"> <Key><Parameter>backend_url</Parameter></Key> </Get> <Get assignTo="kvm.api_key"> <Key><Parameter>third_party_key</Parameter></Key> </Get> </KeyValueMapOperations>

Protocol Transformation

SOAP → REST Conversion

Apigee can expose legacy SOAP/WSDL services as modern REST APIs. Clients call clean JSON/REST endpoints. Apigee translates the call into a SOAP envelope, forwards it to the backend, then converts the XML response back to JSON — completely transparent to the consumer.

Before & After Protocol Bridge
❌ WITHOUT APIGEE
// Client must send SOAP XML POST /WeatherService HTTP/1.1 Content-Type: text/xml SOAPAction: "GetTemperature" <soap:Envelope> <soap:Body> <GetTemperature> <City>Miami</City> </GetTemperature> </soap:Body> </soap:Envelope>
Client tightly coupled to SOAP schema. Any WSDL change breaks clients.
✓ WITH APIGEE PROXY
// Client calls clean REST API GET /v1/weather/Miami HTTP/1.1 Accept: application/json x-api-key: abc123 // Apigee returns clean JSON { "city": "Miami", "temperature": 86, "unit": "F" }
Client is decoupled. SOAP backend can change. REST contract stays stable.
Conversion Pipeline — Step by Step
Client sends REST request
A modern app issues a standard HTTP GET/POST with JSON body or query parameters to the Apigee proxy URL. It knows nothing about SOAP.
Inbound · JSON / REST
WSDL imported → Proxy auto-generated
Apigee reads the WSDL file and automatically generates REST API paths for each SOAP operation. It infers HTTP methods (GET for reads, POST for mutations) and maps query params to SOAP input elements.
Transform · WSDL Parse
ExtractVariables policy — parse REST input
An ExtractVariables policy reads the REST request parameters (path params, query strings, JSON body fields) and stores them as named flow variables in the message context.
Policy · Extract
AssignMessage policy — build SOAP envelope
An AssignMessage policy constructs the full SOAP XML envelope using the extracted variables, sets the Content-Type: text/xml header, and populates the SOAPAction header for the correct operation.
Transform · Build SOAP
Backend receives SOAP — responds with XML
The legacy SOAP service receives a fully-formed SOAP request as if a native SOAP client called it. It processes the request and returns an XML SOAP response.
Backend · SOAP/XML
XMLtoJSON policy — convert response
On the response path, an XMLtoJSON policy automatically parses the SOAP XML response body and transforms it into a clean JSON structure. Namespace prefixes and envelope wrappers are stripped.
Transform · XML → JSON
Client receives clean JSON REST response
The REST client receives a standard JSON response with proper HTTP status codes. An OpenAPI spec is also auto-generated from the WSDL — enabling instant API documentation and SDK generation.
Outbound · JSON / REST
