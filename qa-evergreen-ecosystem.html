01

North Star

Opinionated Decisions

Every design choice below is deliberate. These are the rules of the ecosystem — not suggestions. Deviation requires explicit override, not silence.

Rule 01 · Prerequisite
OpenAPI spec is non-negotiable
Every Java backend must expose an OpenAPI 3.x spec. Auto-generation via springdoc is supported but a quality bar is also enforced: typed DTOs (no raw Object), @Operation(summary=) on all endpoints, explicit nullable flags. A syntactically valid but semantically empty spec generates worthless tests. qa:openapi-check validates both. Poor specs produce a WARNING and a mandatory annotation review Jira item before onboarding proceeds.
enforced: pipeline gate
Rule 02 · Artifact
Breadcrumbs are versioned source files
The .ai-sdlc/qa-breadcrumbs.yaml artifact is committed to the repo, diffed on every MR, and treated like code. Not a side-effect. Not external. Owned by the codebase.
enforced: MR required file check
Rule 03 · Priority
Telemetry beats assumption
When Dynatrace session data exists, journey priority is telemetry-driven. High-traffic paths get comprehensive coverage. Low-traffic paths get smoke level. Static analysis is the fallback, never the preference.
enforced: generator config
Rule 04 · Autonomy
AI fixes, AI reviews, human approves
Self-healing generates a draft PR. AI code review posts a confidence score (40% semantic equivalence + 30% post-patch test pass + 30% blast radius). Locator fixes auto-merge after 24h at ≥90% confidence. Scenario changes always require SDET. Circuit breaker: 3 heals on the same locator in 30 days → Jira escalation, no auto-merge. Regulated repos set QA_AUTOHEALER_AUTOMERGE=false.
enforced: PR workflow rules
Rule 05 · Adoption
One line to adopt, zero to break
Any repo joins the ecosystem by adding one include: line to its .gitlab-ci.yml. The shared template is additive — it never overwrites existing jobs. Feature flags control rollout per repo.
enforced: include template design
Rule 06 · Brownfield
Source code access replaces instrumentation
Brownfield repos are scanned from source — React component tree, Spring controllers, Redux slices, OpenAPI spec. No runtime agent required. No changes to the application. The scanner runs externally and writes only to the QA repo. CMS-built workflow apps (disputes, chargebacks, money movement): the config export is the spec — a cms_config parser strategy reads screens, fields, roles, and workflow states from configuration, more authoritative than parsing generated frontend code.
enforced: scanner design
Rule 07 · Coverage
50% is the floor, not the goal
Every onboarded repo targets 50% as the PoC bar. Metric is explicit: UI coverage = journeys with ≥1 passing scenario ÷ total breadcrumb journeys. API coverage = endpoints with status + schema validation ÷ total OpenAPI endpoints. Pact consumer-only contracts don't count until the provider verifies. Coverage never regresses below the gate.
enforced: coverage gate + audit agent
Rule 08 · Roles
Every role gets its own journey set
Admin, super user, user, and pilot/demo roles each get independently generated journey files. A test tagged @role-admin only runs in contexts where admin access is provisioned. Role drift in the app automatically triggers journey regeneration.
enforced: role-scoped feature files
Rule 09 · Verification
Agents verify in a real browser — async lanes only
The agent-browser CLI closes the gaps static analysis can't see: a per-role verification crawl (snapshot -i a11y tree per route) empirically confirms the breadcrumb page/action/role inventory; intent replay (chat "<gherkin step>") triages locator-break vs real defect before any heal. Fenced with --allowed-domains + --action-policy, role creds from the encrypted auth vault. Cron lanes only — never the blocking MR path. Deterministic Playwright stays the test of record.
enforced: cron verification jobs
Rule 10 · Discovery
Journeys are demonstrated, not imagined
A journey enters the breadcrumbs only with a recorded execution trace proving it's achievable. Three intake lanes: QA-authored plain-English scenarios executed by the agent; autonomous per-role exploration; and the static skeleton that bounds the crawl. A trace compiler freezes each demonstration into a deterministic .feature + steps + POM (locators from a11y role + name — never session refs). api_calls[] is recorded empirically from the HAR. Provenance tracked as discovered_by: static|dynamic|both — undemonstrated journeys don't count toward the coverage denominator. Discovery is expensive once; replay is deterministic Playwright forever.
enforced: trace evidence required in breadcrumb schema

Breadcrumb Schema

The .ai-sdlc/qa-breadcrumbs.yaml is the single source of truth for what the application does, who can do it, and how important it is. Everything downstream — feature files, POM classes, Postman collections — is derived from this file.

How It's Generated — Brownfield vs Greenfield
# breadcrumb_emitter.py # Runs as: python ai-agents/breadcrumb_emitter.py # Reads: source repos (read-only access) # Writes: .ai-sdlc/qa-breadcrumbs.yaml class BreadcrumbEmitter: def emit(self, profile: ProjectProfile): # 1. Detect routing strategy from profile router = RouteParser.for_strategy( profile.routing_strategy # central_routes | file_based | modular ) pages = router.extract_pages( profile.frontend_src ) # 2. Map roles from Spring Security roles = SpringRoleExtractor.extract( profile.backend_src, profile.security_config # reads @PreAuthorize annotations # reads Redux auth slice for UI guards ) # 3. Prioritize via Dynatrace (if available) if profile.dynatrace_available: priority = DynatracePrioritizer\ .rank_pages(pages, api_key=env.DYNATRACE_TOKEN) else: priority = StaticPrioritizer\ .rank_by_depth_and_name(pages) # 4. Map API surface from OpenAPI api = OpenAPIMapper.map( profile.openapi_spec_path ) # 5. Derive journeys per role journeys = JourneyDeriver.derive( pages=pages, roles=roles, priority=priority, api=api, ai_client=ModelGateway() # calls the LLM via the platform gateway # (provider-agnostic — see plan §21.2) # AI synthesizes step sequences # from page graph + action map ) # 6. Write artifact BreadcrumbWriter.write( path='.ai-sdlc/qa-breadcrumbs.yaml', pages=pages, roles=roles, journeys=journeys, api=api, meta=profile.meta ) # 7. Diff against previous version # If journeys removed → flag for retirement # If journeys added → trigger generation DriftDetector.diff_and_emit( prev='.ai-sdlc/qa-breadcrumbs.prev.yaml', curr='.ai-sdlc/qa-breadcrumbs.yaml' )

04

Role-Scoped User Journeys

Journey Derivation

A journey is a named, role-scoped sequence of pages and actions that represents a meaningful unit of business value. Not every permutation — just the ones that matter. Telemetry tells us which ones matter most. The AI deriver proposes candidate journeys; each one must then be demonstrated by the agent-browser discovery lane (Rule 10) before it enters the breadcrumbs — imagined sequences that can't actually be executed never become tests.

Company-Wide GitLab Ruleset

Self-Maintaining System

Autonomy Trust Model

The system acts autonomously within defined trust boundaries. AI fixes and AI reviews. Humans approve. The trust level is proportional to the blast radius of the change.

Trust Boundaries — What the System Does vs What Humans Decide
Locator breaks
e.g. CSS selector stale
AI healer detects failure at runtime. Triage first: agent-browser replays the step's Gherkin intent (chat) — if the intent also fails, it's a real defect → Jira, never heal. If intent succeeds: a11y-tree snapshot finds the new element, patches *Page.ts, draft PR opened. AI posts confidence score (semantic equivalence + post-patch test pass + blast radius). Auto-merge requires both ≥90% confidence and a matching action_renamed drift event — an unexplained DOM change stays red (could be an accidental regression). ≥90% + corroborated → auto-merge after 24h. Circuit breaker: same locator healed 3× in 30 days → Jira escalation, healing suspended. Set QA_AUTOHEALER_AUTOMERGE=false for regulated repos.
Auto-merge 24h · ≥90% confidence
Component renamed
breadcrumb drift detected
Breadcrumb differ detects rename. Re-generates affected POM class and step definitions. Opens draft PR with AI review noting what changed and why. SDET reviews and approves. No auto-merge.
SDET approval
New page added
greenfield or brownfield
Breadcrumb emitter adds new page entry. Generator creates feature file scaffold, step skeleton, POM class. AI reviews generated artifacts for completeness. Draft PR opened. SDET reviews, refines scenarios, approves.
SDET approval
Feature removed
route deleted from source
Drift detector flags orphaned tests. AI marks them @deprecated in the QA repo. Draft PR opened proposing retirement with rationale. SDET must explicitly approve deletion — orphaned tests are never silently removed.
SDET required
Flaky test detected
> 3 intermittent fails
Quarantine agent auto-tags @flaky, moves to quarantine suite, files Jira ticket. Exit requires both: 5 consecutive passing runs in quarantine job AND Jira ticket in Done status. Agent checks both before opening removal PR. Ticket still open = tag stays regardless of pass count.
Fully autonomous · defined exit
Coverage drops
weekly audit
Coverage audit agent runs weekly. Any module below its target gets a generated Jira ticket with the gap analysis, a prioritized backfill list, and draft feature file stubs attached. The pipeline hard-blocks if coverage falls below the configured floor (default 50%).
Hard gate

Retrofit Strategy

Brownfield Onboarding

Existing production repos get full ecosystem coverage without touching their main branch. The scanner reads source code externally. All generated artifacts land in a new QA repo. The feature flag keeps everything parallel until the team is ready to flip.

Brownfield Onboarding Flow — Zero Source Repo DisruptionFeature-Flagged · Parallel Branch
Sprint 0 — Prerequisites Check
Run workspace-analysis on the target repo. Verify: OpenAPI spec present (or auto-generate via springdoc). Read access confirmed to both frontend and backend source. Dynatrace API token available. Create dedicated QA repo: {app-name}-qa.
Day 1–2 · No source repo changes
Sprint 0 — Breadcrumb Generation
Run breadcrumb_emitter.py against the source repo. Outputs .ai-sdlc/qa-breadcrumbs.yaml committed to the QA repo — not the source repo. SDET reviews page inventory and journey derivation for accuracy. Corrections made before proceeding.
Day 2–3 · QA repo only
Week 1 — Generate Initial Test Suite
Generator runs against breadcrumbs. Produces: feature files for all high-priority journeys, POM scaffolds for all mapped pages, API test suite from OpenAPI spec, Postman collection. All land as draft PR in QA repo. AI reviews and scores each artifact. SDET reviews, refines, merges.
Week 1 · QA repo draft PR
Week 2 — Add Ecosystem Include to Source Repo
One line added to the source repo's .gitlab-ci.yml. Variable QA_BROWNFIELD_MODE=true set — this disables the breadcrumb freshness hard gate temporarily. Pipeline now runs QA jobs on every MR but does not block on breadcrumb staleness. Coverage reporting starts.
Week 2 · First pipeline run
Weeks 3–6 — Coverage Climb to 50% Floor
Coverage audit agent runs weekly. Jira tickets auto-filed for uncovered modules. SDET team backfills tests, aided by AI generation. High-priority journeys targeted first. Self-healing activates — any locator breaks from ongoing dev are patched automatically.
Weeks 3–6 · Target 50% floor
Week 6+ — Flip to Full Enforcement
Once 50% floor confirmed: set QA_BROWNFIELD_MODE=false. Breadcrumb freshness gate now enforced. Coverage gate now blocks MRs below floor. Repo is fully integrated. Announce to next team — pattern proven, adoption accelerates.
Week 6 · Full enforcement · Next repo

Adoption Playbook

Repo Onboarding Checklist

The complete checklist for bringing any repo into the ecosystem. Greenfield or brownfield. One team can complete this in a single sprint. The second team takes half the time.

Full Onboarding Checklist — Shareable Per Repo
Prerequisites (Day 1)
  • Read access granted to frontend + backend source repos
  • OpenAPI spec present or springdoc auto-generation confirmed
  • Dynatrace API token provisioned (or static mode accepted)
  • New QA repo created: {app}-qa
  • workspace-analysis skill run, profile committed
  • QA repo bootstrap applied (npm init, playwright, cucumber-js)
Breadcrumb + Generation (Days 2–5)
  • breadcrumb_emitter.py run, YAML committed to QA repo
  • Page inventory reviewed by SDET — gaps noted
  • Journey derivation reviewed — priorities confirmed
  • Generator run — draft PR opened in QA repo
  • AI code review score reviewed by SDET
  • Feature files refined and merged to QA repo main
Pipeline Integration (Week 2)
  • One-line include: added to source repo CI
  • QA_ECOSYSTEM_ENABLED=true set
  • QA_BROWNFIELD_MODE=true set for onboarding window
  • QA_COVERAGE_GATE=50 set as target floor
  • Cron schedules created in GitLab UI (Tue + Thu 06:00)
  • Email distribution list set in QA_REPORT_EMAIL
  • First MR pipeline run verified — QA jobs visible
Go-Live Gate (Week 6)
  • 50% API coverage confirmed in coverage-manifest.json
  • 50% UI coverage confirmed in allure report
  • Self-healing confirmed on at least one locator fix
  • First regression email report delivered successfully
  • QA_BROWNFIELD_MODE=false — full enforcement active
  • Retrospective held — lessons fed back to ecosystem template
Definition of Done — Per Repo
50% API coverage 50% UI coverage breadcrumbs committed CI include active regression email live self-heal verified coverage gate enforced next team nominated
